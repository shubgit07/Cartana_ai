VoiceTask AI — MVP Documentation
Version: 1.0
Status: Based on current partially implemented codebase

1. Project Overview

VoiceTask AI is a multimodal task-extraction system that converts managerial voice notes or text into structured tasks. The MVP focuses on a working end-to-end pipeline: input ingestion → async processing → transcription or text handling → task detection and entity extraction → normalization → persistence → UI display in a Kanban board.

This MVP is intentionally designed as a product-ready foundation, not the final production system. The final long-term direction should remain aligned with the full architecture described in the project specification, which includes a modular AI pipeline, async worker orchestration, deterministic normalization, and human-in-the-loop task editing. 
2. MVP Goal

The MVP must prove one core outcome:

A user can submit a voice note or text note, the system extracts actionable tasks asynchronously, and those tasks appear in the UI where they can be reviewed, edited, moved, and deleted.

The MVP should validate:
- voice-to-task extraction works reliably enough for demos and internal use
- async processing behaves correctly with job status polling
- extracted tasks are normalized into usable fields
- the UI can show and manage tasks in a simple Kanban workflow

4. MVP Scope

4.1 In Scope

The MVP includes:

- S3/object storage integration

5. Product Behavior

The MVP should work as follows:

1. A user records a voice note or submits text.
2. The backend creates a Note record and a job_id.
3. The async worker processes the note.
4. If audio was submitted, it is transcribed to text.
5. The system detects action items from the transcript.
6. Each action item is converted into structured task fields.
7. Deadlines and priorities are normalized.
8. Tasks are saved in PostgreSQL.
9. The frontend polls job status and refreshes automatically.
10. The user sees extracted tasks in the Kanban board and can edit them.

7. Backend Components

7.1 Data Models

User
- id
- username
- role
- relationship to tasks

Note
- id
- job_id (UUID)
- raw_text
- status: pending / processed / failed
- pipeline_trace (JSONB)

Task
- id
- job_id (UUID)
- title
- description
- priority
- deadline
- status: TODO / IN_PROGRESS / DONE
- assignee_id
- note_id

7.2 API Endpoints

POST /process-input
- accepts text or audio file
- creates Note
- dispatches Celery job
- returns job_id and note_id

GET /status/{job_id}
- returns job state: PENDING / SUCCESS / FAILURE

GET /notes/{note_id}
- returns note details, pipeline trace, and related tasks

GET /tasks
- returns task list
- supports optional status filter

PATCH /tasks/{task_id}
- allows updates to task fields

DELETE /tasks/{task_id}
- removes a task

8. AI Processing Pipeline

The MVP pipeline should be implemented as separate steps so each step is testable and replaceable.

8.1 Transcription
- Input: uploaded audio
- Output: transcript text
- Model/service: Groq Whisper Large V3

8.2 Task Detection
- Input: transcript
- Output: list of action sentences
- Model/service: Groq LLM call
- Purpose: identify only the sentences that represent actionable work

8.3 Entity Extraction
- Input: each action sentence
- Output:
  - title
  - description
  - assignee_name
  - priority
  - deadline_text
- Purpose: convert plain language into structured task metadata

8.4 Normalization
- Deadline normalization: convert natural language dates into ISO date format
- Priority normalization: map informal priority phrases to high / medium / low

8.5 Persistence
- Save tasks to DB
- Store pipeline trace in JSONB
- Mark note as processed or failed

9. Normalization Rules

9.1 Date Normalization
Use timezone-aware parsing for Asia/Kolkata.

Examples:
- tomorrow → resolved date
- next Friday → nearest future Friday
- April 2 → ISO date
- no deadline → null

9.2 Priority Normalization
Map informal language into structured priority:

High
- urgent
- asap
- immediately
- today

Medium
- soon
- important
- next week

Low
- whenever
- no rush
- sometime

11. MVP User Stories

Story 1
As a manager, I can record a voice note and get tasks automatically extracted.

Story 2
As a manager, I can submit typed text and get the same task extraction behavior.

Story 3
As a user, I can see whether processing is pending, running, succeeded, or failed.

Story 4
As a user, I can review the AI output before trusting it.

Story 5
As a user, I can edit, move, and delete tasks after extraction.

12. Acceptance Criteria for MVP

The MVP is complete when all of the following are true:
- audio input can be recorded and submitted
- text input can be submitted through the backend
- a note is created for each submission
- Celery worker processes the note asynchronously
- transcript is generated for audio input
- action items are detected from transcript
- structured tasks are extracted and normalized
- tasks are stored in PostgreSQL
- frontend polls job status successfully
- tasks appear in the Kanban board automatically
- inline edit and delete work
- pipeline trace is visible
- no manual refresh is needed after a successful run

3. Ensure task persistence and task listing are correct.

6. Improve pipeline trace readability.

15. Non-Functional Requirements for MVP

Even at MVP stage, the system should be:
- reliable enough for repeated demos
- deterministic where possible
- easy to debug using pipeline trace
- modular enough for future replacement of AI components
- simple enough to maintain while developing the next version
